## ADRs (locked)

### ADR-1: Simplified single-step visa
One approver per document, not parallel multi-approver routing. Tracked via
`visa_signed_by` + `visa_signed_at` audit columns. Multi-step routing requires
separate `document_routing_steps` table — deferred.

**Rationale**: Defense scope; diploma demonstrates end-to-end lifecycle, not
production-grade routing engine. Adding multi-step now puts PR over 1000 LOC.

### ADR-2: Admin-only role gate for both transitions
Both `SendToRouting` and `SignVisa` mounted on the existing `/admin/documents`
group (gated by `RequireRole(AcademicSecretary, SystemAdmin)`). No separate
role for "visa approver" — academic secretary signs visa per defense flow.

**Rationale**: Aligns with v0.148.0 ADR-1 (Methodist NOT admin gate).
Avoids new role enum value mid-phase-pack.

### ADR-3: ErrCannotRoute on non-registered + ErrCannotSignVisa on non-routing
Status invariant guards mirror Register/Approve/Reject pattern. No combined
`ErrCannotTransition` umbrella — per-transition sentinels keep audit reason
strings unambiguous (`not_registered` / `not_routing`).

### ADR-4: Two use cases, two handlers — не один с verb param
`StartRoutingUseCase` + `SignVisaUseCase` separate types, NOT a single
`TransitionWorkflowUseCase` with verb enum. Mirrors all 5 existing use cases
(Submit/Approve/Reject/Register) — pattern consistency over DRY.

**Rationale**: Each transition has different audit action name + future
phases may add per-transition params (e.g. visa comment, conditional approval).

### ADR-5: Frontend dialogs без input field for both transitions
SendToRouting and SignVisa are confirmation-only — no required input (visa
comment is out of scope per ADR-1). Match Submit/Approve dialog shape, not
Register/Reject (which have input).

### ADR-6: i18n keys reuse `documentsWorkflow` namespace
New keys: `routing.*`, `signVisa.*`, `routingToast.*`, `signVisaToast.*`,
`actions.routeButton`, `actions.signVisaButton`. statusBadge already has
`routing` + `execution` (added в v0.148.0 pre-emptively).

### ADR-7: T1-A verify upfront — `docSelectCols` updated в same PG repo commit
Per `feedback_reviewer_tier1_workflow_classes` — when migration adds columns,
PG Update / GetByID / List + `docSelectCols` test helper extend в the same
commit. Block class observed в v0.148.0 round-1 reviewer fix-cycle.

### ADR-8: T1-B handler envelope = `response.Success(doc)`
Match Register endpoint (v0.149.0) — `c.JSON(http.StatusOK, response.Success(doc))`.
Frontend hook unwraps `.data`. Block class observed в v0.148.0 round-1.
