## 1. Введение и обоснование

### 1.1 Постановка проблемы

Информационная система академического секретаря-методиста обслуживает до 5000 пользователей и автоматизирует документооборот, управление расписанием, отслеживание успеваемости и внутренние коммуникации образовательного учреждения. Ряд процессов в системе может быть существенно улучшен за счёт внедрения технологий машинного обучения и нейронных сетей:

- **Предиктивная аналитика** — текущая rule-based система оценки рисков студентов использует жёсткие пороговые значения, что не учитывает нелинейные зависимости и индивидуальные паттерны поведения.
- **Поиск по документам** — полнотекстовый поиск не учитывает семантику запроса, что снижает релевантность результатов.
- **Классификация и обработка документов** — ручная категоризация входящих документов и извлечение ключевых полей занимает значительное время.
- **Составление расписания** — ручная оптимизация не гарантирует оптимальное распределение ресурсов.
- **Рутинные операции** — повторяющиеся задачи секретаря могут быть автоматизированы через интеллектуального ассистента.

### 1.2 Принципы выбора технологий

При выборе моделей и инструментов были приняты следующие принципы:

1. **Полностью открытый исходный код** — все используемые модели и инструменты имеют открытые лицензии (Apache 2.0, MIT, BSD).
2. **Локальное развёртывание** — данные не покидают периметр организации, что критично для работы с персональными данными студентов и сотрудников (152-ФЗ).
3. **Нулевые затраты на лицензии и API** — отсутствие зависимости от платных облачных API.
4. **Поддержка русского языка** — все модели проверены на бенчмарках для русского языка.
5. **Интеграция с существующим стеком** — совместимость с Go-бэкендом, PostgreSQL и существующей архитектурой.

## 3. Подсистема RAG: интеллектуальный поиск по документам

### 3.1 Текущая реализация

Система Retrieval-Augmented Generation (RAG) позволяет пользователям задавать вопросы по содержимому загруженных документов на естественном языке:

- **Чанкинг**: документы разбиваются на фрагменты по 512 токенов с перекрытием 10%.
- **Эмбеддинги**: текущая реализация использует OpenAI `text-embedding-3-small` (1536 измерений).
- **Индекс**: pgvector IVFFlat (100 списков) для приближённого поиска ближайших соседей.
- **Порог релевантности**: cosine similarity >= 0.7.
- **LLM**: OpenAI `gpt-4o-mini` для генерации ответов с цитированием источников.

### 3.2 Целевая реализация: полностью локальный RAG

#### Модель эмбеддингов: BGE-M3

**Выбранная модель**: [BAAI/bge-m3](https://github.com/FlagOpen/FlagEmbedding).

| Параметр | Значение |
|----------|----------|
| Размерность вектора | 1024 |
| Поддержка языков | 100+ (включая русский) |
| Размер модели | 567 MB |
| Лицензия | MIT |

**Обоснование выбора на основе бенчмарка ruMTEB:**

Бенчмарк ruMTEB (Russian Massive Text Embedding Benchmark) — специализированный набор задач для оценки моделей эмбеддингов на русском языке, включающий семантическое сходство, классификацию, реранжирование и поиск (Snegirev et al., 2024, NAACL 2025).

| Модель | ruMTEB (avg) | Retrieval | Размер | Локальная | Стоимость |
|--------|-------------|-----------|--------|-----------|-----------|
| E5-mistral-7b-instruct | 69.11 | Высокий | 14 GB | Да | $0 |
| **BGE-M3** | **60.46** | **Лучший** | **567 MB** | **Да** | **$0** |
| ru-en-RoSBERTa | 62.74 | Средний | 500 MB | Да | $0 |
| nomic-embed-text | ~55-58 | Средний | 274 MB | Да | $0 |
| OpenAI text-embedding-3-small | ~62 | Высокий | — | Нет | ~$5/мес |

BGE-M3 занимает лучшие позиции по задачам Retrieval и Reranking — ключевым для RAG-системы, при этом значительно компактнее E5-mistral (567 MB vs 14 GB).

**Альтернативная модель (лёгкая)**: `nomic-embed-text` (274 MB) — для серверов с ограниченной оперативной памятью. Качество ниже, но скорость инференса выше.

#### Миграция с OpenAI на локальные эмбеддинги

Миграция требует следующих шагов:

1. Изменение размерности вектора в pgvector с 1536 на 1024.
2. Переключение провайдера эмбеддингов на Ollama в конфигурации.
3. Полная переиндексация всех документов.

```sql
-- Миграция pgvector: изменение размерности
ALTER TABLE ai_embeddings
  ALTER COLUMN embedding TYPE vector(1024);

-- Пересоздание IVFFlat индекса
DROP INDEX IF EXISTS idx_ai_embeddings_vector;
CREATE INDEX idx_ai_embeddings_vector
  ON ai_embeddings USING ivfflat (embedding vector_cosine_ops)
  WITH (lists = 100);
```

Текущая реализация эмбеддингов использует OpenAI API (`text-embedding-3-small`). Миграция на BGE-M3 — отдельный этап.

### 3.3 Метрики оценки RAG-подсистемы

| Метрика | Описание | Целевое значение |
|---------|----------|------------------|
| Retrieval Precision@5 | Доля релевантных документов в топ-5 | >= 0.75 |
| Retrieval Recall@10 | Полнота поиска в топ-10 | >= 0.85 |
| Answer Faithfulness | Точность ответа относительно источников | >= 0.80 |
| Латентность (эмбеддинг) | Время генерации эмбеддинга одного чанка | < 100 мс |
| Латентность (поиск) | Время поиска по pgvector | < 50 мс |

## 4. Подсистема предиктивной аналитики

### 4.1 Текущая реализация (rule-based)

Система определяет уровень риска студента по жёстким пороговым значениям:

| Уровень | Критерии |
|---------|----------|
| Critical | Посещаемость < 60% ИЛИ Средний балл < 50 |
| High | Посещаемость < 75% ИЛИ Средний балл < 65 |
| Medium | Посещаемость < 85% ИЛИ Средний балл < 75 |
| Low | Все показатели в норме |

**Недостатки текущего подхода:**
- Не учитывает нелинейные зависимости между признаками.
- Не адаптируется к конкретному контексту (специальность, курс, семестр).
- Одинаковые пороги для всех студентов, независимо от индивидуальных паттернов.
- Не использует временные тренды для прогнозирования.

### 4.2 Целевая реализация: ML-модель на базе градиентного бустинга

#### Выбор алгоритма: XGBoost / LightGBM

| Параметр | XGBoost | LightGBM | Random Forest |
|----------|---------|----------|---------------|
| Качество на табличных данных | Отличное | Отличное | Хорошее |
| Скорость обучения | Высокая | Очень высокая | Средняя |
| Интерпретируемость | Высокая (SHAP) | Высокая (SHAP) | Средняя |
| Feature importance | Да | Да | Да |
| Работа с пропусками | Нативная | Нативная | Требует обработки |
| Лицензия | Apache 2.0 | MIT | BSD (scikit-learn) |

**Рекомендация**: XGBoost как основная модель, логистическая регрессия как baseline для сравнения.

#### Вектор признаков

На основании данных, доступных в системе, формируется следующий вектор признаков:

| Признак | Источник | Тип | Описание |
|---------|----------|-----|----------|
| `attendance_rate` | `v_student_attendance_stats` | float | Процент посещаемости (0-100) |
| `avg_grade` | `v_student_grade_stats` | float | Средневзвешенная оценка |
| `attendance_trend_4w` | `v_monthly_attendance_trend` | float | Изменение посещаемости за 4 недели |
| `grade_trend_4w` | `grades` | float | Изменение среднего балла за 4 недели |
| `consecutive_absences` | `attendance_records` | int | Макс. серия подряд идущих пропусков |
| `late_ratio` | `attendance_records` | float | Доля опозданий среди посещений |
| `homework_completion` | `grades` (type=homework) | float | % выполненных домашних заданий |
| `exam_avg` | `grades` (type=midterm/final) | float | Средний балл за контрольные |
| `course_number` | `external_students` | int | Курс обучения (1-6) |
| `semester_week` | вычисляемый | int | Номер недели семестра (1-18) |
| `lessons_per_week` | `lessons` | int | Количество занятий в неделю |
| `subjects_count` | `lessons` | int | Количество дисциплин |

**Целевая переменная**: `risk_level` — категориальная (low / medium / high / critical), определённая экспертной разметкой или ретроспективным анализом (отчислен / переведён / оставлен на повторное обучение).

#### Архитектура инференса

```
┌────────────────────────────────────────────────────────┐
│              Обучение (офлайн, Python)                  │
│                                                        │
│  PostgreSQL → pandas DataFrame → Feature Engineering   │
│       → XGBoost.fit() → model.json (экспорт)          │
│       → SHAP values (интерпретация)                    │
└────────────────────┬───────────────────────────────────┘
                     │ model.json
                     ▼
┌────────────────────────────────────────────────────────┐
│              Инференс (онлайн, Go)                     │
│                                                        │
│  Go Backend → SQL (фичи) → leaves.XGEnsembleFromFile  │
│       → prediction → risk_level + probability          │
│       → student_risk_assessments (кеш)                 │
└────────────────────────────────────────────────────────┘
```

**Обучение** выполняется периодически (1 раз в семестр или при накоплении достаточного объёма новых данных) на Python с использованием XGBoost и scikit-learn.

**Инференс** выполняется в Go-бэкенде через библиотеку [`dmitryikh/leaves`](https://github.com/dmitryikh/leaves) — чистая Go-реализация предсказания для GBRT-моделей, поддерживающая загрузку XGBoost и LightGBM моделей без внешних зависимостей.

```go
import "github.com/dmitryikh/leaves"

// Загрузка обученной модели
model, err := leaves.XGEnsembleFromFile("models/risk_model.json", false)

// Предсказание для студента
features := []float64{
    attendanceRate, avgGrade, attendanceTrend,
    gradeTrend, consecutiveAbsences, lateRatio,
    homeworkCompletion, examAvg, courseNumber,
    semesterWeek, lessonsPerWeek, subjectsCount,
}
predictions := make([]float64, model.NOutputGroups())
model.PredictSingle(features, predictions)
// predictions содержит вероятности для каждого класса риска
```

### 4.3 Валидация модели

| Метрика | Описание | Целевое значение |
|---------|----------|------------------|
| Accuracy | Общая точность классификации | >= 0.80 |
| Recall (Critical) | Полнота выявления критических случаев | >= 0.90 |
| Precision (Critical) | Точность предсказания критического риска | >= 0.75 |
| F1-macro | Среднее F1 по всем классам | >= 0.78 |

Валидация проводится методом stratified k-fold cross-validation (k=5) с сохранением пропорций классов.

## 6. NLP-подсистема обработки документов

### 6.1 Классификация документов

#### Модель: rubert-tiny2

**Выбранная модель**: [`cointegrated/rubert-tiny2`](https://huggingface.co/cointegrated/rubert-tiny2)

| Параметр | Значение |
|----------|----------|
| Параметры | 29M |
| Размер | ~120 MB |
| Архитектура | BERT (6 слоёв, 312 hidden) |
| Лицензия | MIT |
| Язык | Русский (основной), английский |

**Задача**: автоматическая классификация входящих документов по типам (приказ, служебная записка, протокол, договор, учебный план, отчёт и др.).

**Подход**: fine-tuning предобученной модели на размеченном корпусе документов организации. Минимальный объём обучающей выборки — 50-100 документов каждого типа.

```python
from transformers import AutoTokenizer, AutoModelForSequenceClassification

model_name = "cointegrated/rubert-tiny2"
tokenizer = AutoTokenizer.from_pretrained(model_name)
model = AutoModelForSequenceClassification.from_pretrained(
    model_name, num_labels=10  # количество типов документов
)
# Fine-tuning на корпусе организации...
```

### 6.2 Извлечение именованных сущностей (NER)

#### Модель: DeepPavlov NER

**Выбранная модель**: [`DeepPavlov/ner_rus_bert`](https://github.com/deeppavlov/DeepPavlov)

| Параметр | Значение |
|----------|----------|
| Архитектура | BERT + CRF |
| Сущности | PER, ORG, LOC, DATE (расширяемо) |
| Качество (F1) | ~0.90 на русском |
| Размер | ~700 MB |
| Развёртывание | Docker, Python API |
| Лицензия | Apache 2.0 |

**Задача**: автоматическое извлечение ключевых полей из текста документов для заполнения шаблонов:

| Тип сущности | Пример | Применение в системе |
|-------------|--------|---------------------|

## 7. Подсистема оптимизации расписания

### 7.1 Постановка задачи

Задача составления расписания учебного заведения является задачей удовлетворения ограничений (Constraint Satisfaction Problem, CSP) и относится к NP-трудным задачам.

**Входные данные:**
- Группы студентов (списки, специальности, курс).
- Преподаватели (специализации, предпочтения, ограничения по времени).
- Аудитории (вместимость, оборудование).
- Дисциплины (требования к аудиториям, длительность, тип занятия).
- Учебные планы (количество часов по дисциплинам).

**Жёсткие ограничения (hard constraints):**
- Преподаватель не может вести два занятия одновременно.
- Группа не может посещать два занятия одновременно.
- Аудитория не может быть занята двумя группами.
- Вместимость аудитории >= размер группы.
- Соблюдение нагрузки преподавателя (не более N часов в день/неделю).

**Мягкие ограничения (soft constraints, оптимизация):**
- Минимизация "окон" в расписании студентов.
- Предпочтения преподавателей по времени.
- Равномерное распределение нагрузки по дням недели.
- Минимизация перемещений между корпусами.

### 7.2 Выбор метода: Google OR-Tools CP-SAT

**Выбранный инструмент**: [Google OR-Tools](https://developers.google.com/optimization) (CP-SAT Solver)

| Параметр | Значение |
|----------|----------|
| Тип | Constraint Programming (CP) |
| Солвер | CP-SAT (Satisfiability Solver) |
| Лицензия | Apache 2.0 |
| Языки | Python, C++, Java, C# |
| Стоимость | Бесплатно |

**Обоснование:**
- Специализирован для задач планирования и составления расписаний.
- Поддержка жёстких и мягких ограничений с приоритетами.
- Масштабируемость до тысяч переменных.
- Активно поддерживается Google.
- Существуют готовые примеры для [университетских расписаний](https://github.com/giangstrider/scheduling-optimization-ortools).

### 7.3 Архитектура подсистемы

```
┌─────────────────────────────────────────────────────────┐
│  Сервис оптимизации расписания (Python + OR-Tools)      │
│                                                         │
│  POST /optimize    → запуск оптимизации                 │
│  GET  /status/:id  → статус выполнения                  │
│  GET  /result/:id  → получение результата               │
│                                                         │
│  Алгоритм:                                              │
│  1. Загрузка данных (группы, преподаватели, аудитории)  │
│  2. Формулировка ограничений (CP-SAT model)             │
│  3. Решение (с таймаутом и лучшим найденным решением)  │
│  4. Возврат расписания в формате JSON                    │
└─────────────────────────────────────────────────────────┘
```

## 12. Оценка рисков
